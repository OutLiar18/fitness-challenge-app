# Champions Legacy

# Version History

Version: 2.0

---

# Purpose

Version History defines the versioning strategy used throughout Champions Legacy.

A consistent versioning system makes it easier to understand the maturity of the project, track progress over time and identify when significant milestones have been reached.

Every public release should be assigned a version number following the guidelines in this document.

---

# Version Format

Champions Legacy uses Semantic Versioning (SemVer):

MAJOR.MINOR.PATCH

Example:

1.4.2

Where:

- **MAJOR** – Breaking changes or significant project milestones.
- **MINOR** – New features and functionality that remain backwards compatible.
- **PATCH** – Bug fixes, performance improvements and small refinements.

---

# Major Versions

Increase the **MAJOR** version when:

- A major architectural change is introduced.
- Breaking changes are made.
- A significant project milestone has been reached.
- A production-ready version is released.

Examples:

0.0.0 → 1.0.0

1.0.0 → 2.0.0

Major releases should represent meaningful progress.

---

# Minor Versions

Increase the **MINOR** version when:

- New features are added.
- Existing functionality is expanded.
- New modules are introduced.
- Significant improvements are made without breaking compatibility.

Examples:

0.2.0 → 0.3.0

1.4.0 → 1.5.0

Most development will occur through minor version updates.

---

# Patch Versions

Increase the **PATCH** version when:

- Bugs are fixed.
- Performance is improved.
- Minor UI improvements are made.
- Documentation corrections accompany small implementation changes.

Examples:

0.3.0 → 0.3.1

1.5.2 → 1.5.3

Patch releases should not introduce major new functionality.

---

# Pre-1.0 Development

Until Champions Legacy reaches Version **1.0.0**, the project is considered to be in active development.

During this stage:

- Features may change significantly.
- Architecture may continue to evolve.
- Systems may be refactored as needed.
- Feedback should guide development.

Version 1.0.0 represents the first stable production-ready release.

---

# Release Milestones

The following milestones provide a general guide for the project's progression.

## 0.x.x

Active development.

Core systems are being designed, implemented and refined.

---

## 1.0.0

First stable production release.

Core functionality is complete and suitable for real-world challenges.

---

## 2.x.x+

Major platform evolution.

Examples may include:

- Expanded challenge systems.
- Advanced analytics.
- Team competitions.
- Administration tools.
- New platform capabilities.

---

# Versioning Guidelines

When creating a new version:

- Update this document if the versioning strategy changes.
- Add an entry to the Changelog.
- Create Release Notes for significant releases.
- Tag the release in Git (when appropriate).

Version numbers should remain meaningful and reflect genuine progress.
