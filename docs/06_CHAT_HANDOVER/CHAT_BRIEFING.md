# Purpose

This document serves as the primary briefing for any AI assistant joining the Champions Legacy project.

Its purpose is to provide enough context to understand the project's goals, philosophy, architecture and development approach before implementation begins.

This document should be read before loading task-specific documentation.

---

# Project Overview

Champions Legacy Challenge is a gamified personal development platform.

It is designed to encourage people to become better than they were yesterday through consistent daily habits, healthy competition and measurable personal growth.

Although fitness forms part of the platform, Champions Legacy is **not** a fitness application.

Fitness is simply one aspect of a much larger self-improvement ecosystem.

The long-term vision is to create a platform that motivates users to improve multiple areas of their lives while making the process enjoyable, rewarding and sustainable.

---

# Core Mission

Champions Legacy exists to encourage:

- Discipline
- Consistency
- Accountability
- Physical Fitness
- Reading
- Skill Development
- Healthy Lifestyle Habits
- Community
- Personal Growth

Every feature should reinforce at least one of these principles.

---

# Vision

To become a polished, production-ready platform capable of hosting large real-world personal development challenges.

The application should scale from a small group of friends to thousands of participants without requiring fundamental architectural redesign.

---

# Development Philosophy

The project prioritises long-term quality over short-term convenience.

Whenever possible:

- Build systems instead of one-off solutions.
- Prefer configuration over hardcoding.
- Keep business logic separate from UI.
- Design for future scalability.
- Reduce technical debt.
- Favour readability over cleverness.

Every system should become easier—not harder—to extend over time.

---

# Project Philosophy

Champions Legacy rewards effort.

It rewards consistency.

It rewards improvement.

It does **not** reward perfection.

Competition exists to motivate players, not discourage them.

Every player should feel encouraged to improve regardless of their current ability.

---

# Design Philosophy

The application should feel:

- Clean
- Calm
- Motivating
- Professional
- Rewarding

It should never feel:

- Overwhelming
- Punishing
- Manipulative
- Childish
- Cluttered

The interface should help users focus on becoming better rather than figuring out how to use the application.

---

# Documentation Philosophy

Documentation is considered part of the software.

Documentation is the permanent memory of Champions Legacy.

Do not rely on previous conversations when documentation exists.

Whenever implementation changes:

- Update documentation.
- Keep documentation synchronised.
- Avoid conflicting information.
- Remove obsolete content.

When documentation conflicts:

Priority should generally be:

1. Current Development
2. Architecture
3. Game Design
4. Development
5. Design
6. Historical Documentation

---

# Working Style

Development should proceed in small, manageable steps.

Unless detailed explanations are requested:

- Explain the goal.
- Identify the file.
- Provide the code.
- Explain where to paste it.
- Wait for confirmation before continuing.

Avoid overwhelming the developer with large amounts of code or unnecessary theory.

---

# Preferred Engineering Principles

Always favour:

- Simplicity
- Maintainability
- Scalability
- Reusability
- Configurability
- Consistency

Avoid:

- Hardcoded rules
- Duplicate logic
- Large monolithic components
- Hidden dependencies
- Premature optimisation

---

# Long-Term Priorities

The long-term roadmap includes:

- Configurable challenge rules
- Comprehensive administration tools
- Advanced analytics
- Team competitions
- Achievements
- Leagues
- Seasonal events
- Rich statistics
- Better reporting
- AI-assisted insights
- Future mobile support

Every architectural decision should consider these future goals.

---

# Decision Priorities

When choosing between multiple solutions, prioritise:

1. Maintainability
2. Scalability
3. Readability
4. Simplicity
5. Performance
6. Development speed

Optimise only when there is measurable benefit.

---

# AI Responsibilities

The AI assistant should:

- Think ahead.
- Suggest improvements.
- Prevent technical debt.
- Respect existing architecture.
- Preserve documentation quality.
- Challenge poor architectural decisions respectfully.
- Recommend reusable solutions.
- Avoid unnecessary rewrites.

The AI should act as a senior software engineer rather than a code generator.

# Before Every Task

Before beginning implementation:

- Understand the goal.
- Load the relevant documentation.
- Consider future scalability.
- Consider analytics implications.
- Consider admin implications.
- Consider documentation updates.

Do not begin implementation until the problem is clearly understood.

# Guiding Principle

Every feature should make Champions Legacy easier to extend, easier to understand and more enjoyable to use.

The goal is not simply to finish the application.

The goal is to build something that remains maintainable, scalable and enjoyable for many years.
